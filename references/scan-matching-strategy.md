# 扫描件匹配策略

`ocr_scans.py` 内置 3 阶段 Pipeline。源码见 [ocr_scans.py](../scripts/ocr_scans.py)。

## Phase 1：快速扫（~60% 命中）

- psm=6, 1x 不反色，每个文件一次 Tesseract
- 发票号精确匹配 / 尾 15 位模糊匹配 → 行
- 金额 ±2 元最近邻匹配 → 行
- 命中后不再参与后续阶段

## Phase 2：深搜（覆盖至 ~80%）

仅对 Phase 1 未命中的文件：
- 2x 放大 + autocontrast + sharpen
- psm=[3, 6, 7]，正常 + 反色各跑一次
- 同样发票号优先、金额兜底

## Phase 3：消元法（覆盖至 100%）

前两阶段仍无法匹配的，按行需求分配：

1. 统计每行已有发票和行程单数量
2. 打车行先补发票位，再补行程单位；高铁/住宿只补发票位
3. **不依赖 OCR 关键词分类**——任何类型未分配扫描件均可补位
4. 时序保证：发票位永远先于行程单位，避免行程单被误当发票

**消元法设计要点：**
- Tesseract 对英文关键词（`DIDI TRAVEL`）输出不稳定，单字符 `'D'`/`'G'` 会误命中 `train_kw`。因此消元法阶段不使用 `classify_scan` 做类型过滤。
- 如果 OCR 已经给出了发票号精确匹配（发票），消元法优先把它们分配给发票位而非行程单位。

## 无法分配

剩余孤儿标记 WARN，不阻止生成。常见原因：
- 扫描件是备份/重拍，对应行已有足够素材
- 扫描件内容过于模糊，所有阶段无法读取
